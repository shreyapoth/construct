### Phase Overview

**Purpose:** Convert approved requirements and guidelines into a concrete execution plan that can be implemented with low ambiguity.

**Key Principle:** Plan the work before doing the work.

This phase establishes:

- **Execution approach** — the implementation strategy and sequencing
- **Work breakdown** — clear steps or work units with dependencies
- **Validation plan** — how each major step will be verified
- **Checklist** — a trackable list of tasks to execute and monitor progress

The plan should be specific enough that execution can begin without guessing, while still allowing updates if new facts emerge.

---

### Rules

- Build the plan from the **approved outputs of Phase 0 and Phase 1** (goal, requirements, constraints, scope, open questions).
- Translate requirements into **ordered, actionable steps**; avoid vague items like “implement feature.”
- Include **dependencies, risks, and assumptions** where they affect order or correctness.
- Define **verification** for major steps (tests, checks, acceptance criteria mapping, or review points).
- Create and maintain a **checklist** for the execution phase; each item should be concrete and checkable.
- If there are unresolved questions that block correctness, stop and surface them before finalizing the plan.
- If the plan changes materially, summarize what changed and why before continuing.
- **Do not** move on until the user **explicitly approves** this phase: briefly recap the implementation plan and checklist, then ask if they are ready to continue (they may revise first).
